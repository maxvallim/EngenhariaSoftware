Diagrama de Sequência como Ponte para Arquiteturas

Nesta aula utilizamos o Diagrama de Sequência como base para compreender arquiteturas de software, especialmente MVC, MVP e MVVM.

Objetivos da Aula

  • Revisar o diagrama de sequência
  • Relacionar fluxo de mensagens com arquitetura
  • Entender MVC na prática
  • Comparar MVC, MVP e MVVM
  • Identificar boas e más práticas

1. Diagrama de Sequência

O diagrama de sequência representa o comportamento dinâmico de um sistema, destacando a ordem das interações entre objetos.

Pergunta-chave:
Quem conversa com quem para realizar um caso de uso?

Exemplo 1 – Cadastrar Usuário (sem arquitetura definida)

@startuml
actor Usuario

participant "TelaCadastro" as Tela
participant "Sistema" as Sistema
participant "BancoDados" as BD

Usuario -> Tela : preencherDados()
Tela -> Sistema : enviarDados(dados)
Sistema -> BD : insertUsuario(dados)
BD --> Sistema : OK
Sistema --> Tela : sucesso()
Tela --> Usuario : mensagemSucesso()
@enduml
  • O que é o objeto Sistema?
  • Onde está a regra de negócio?
  • Esse modelo é sustentável?

2. Arquitetura e Organização do Fluxo

Arquitetura define responsabilidades e limita quem pode se comunicar com quem. Isso aparece claramente no diagrama de sequência.

Um diagrama de sequência bem feito revela a arquitetura em funcionamento.

3. MVC no Diagrama de Sequência

  • View: interface
  • Controller: controle do fluxo
  • Model: regra de negócio e dados

Exemplo 2 – Cadastrar Usuário com MVC

@startuml
actor Usuario

participant "View\n(TelaCadastro)" as View
participant "Controller\n(UsuarioController)" as Controller
participant "Model\n(UsuarioService)" as Model
participant "Repository\n(UsuarioRepository)" as Repo
participant "Banco de Dados" as BD

Usuario -> View : preencherDados()
View -> Controller : cadastrarUsuario(dados)
Controller -> Model : validarECriarUsuario(dados)
Model -> Repo : salvar(usuario)
Repo -> BD : INSERT usuario
BD --> Repo : OK
Repo --> Model : sucesso
Model --> Controller : usuarioCriado
Controller --> View : exibirSucesso()
@enduml
  • Separação clara de responsabilidades
  • Baixo acoplamento
  • Facilidade de manutenção

4. Exemplo de Anti‑Padrão

Violação de arquitetura: a interface acessando diretamente o banco de dados. 
@startuml
actor Usuario
participant "TelaCadastro" as View
participant "Banco de Dados" as BD

Usuario -> View : preencherDados()
View -> BD : INSERT usuario
BD --> View : OK
View --> Usuario : sucesso()
@enduml
  • Alto acoplamento
  • Dificuldade de testes
  • Manutenção comprometida

5. Comparação de Arquiteturas

MVP – Model View Presenter

@startuml
actor Usuario

participant "View" as View
participant "Presenter" as Presenter
participant "Service" as Model

Usuario -> View : clicarSalvar()
View -> Presenter : onSalvar(dados)
Presenter -> Model : cadastrarUsuario(dados)
Model --> Presenter : sucesso
Presenter -> View : mostrarMensagem()
@enduml

MVVM – Model View ViewModel

@startuml
actor Usuario

participant "View" as View
participant "ViewModel" as VM
participant "Model" as Model

Usuario -> View : interagir()
View -> VM : atualizarPropriedade()
VM -> Model : processarDados()
Model --> VM : resultado
VM --> View : notificarMudanca()
@enduml
