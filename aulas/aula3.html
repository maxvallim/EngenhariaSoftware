Aula 03 — Requisitos (RF/RNF), Regras de Negócio e GitHub (Issues, Labels, Milestones, PR)

  • Escrever requisitos funcionais, não funcionais e regras de negócio com exemplos claros.
  • Transformar requisitos em Issues e organizar com Labels e Milestones.
  • Abrir Pull Requests com boas descrições e fechar Issues automaticamente (Closes #).

1) Conceitos essenciais

Requisito Funcional (RF)

O que o sistema deve fazer.

RF01 — Login do usuário
O sistema deve permitir login com e-mail e senha.

Anti-exemplo: "Tela bonita" (isso é qualidade/UX, não funcionalidade).

Requisito Não Funcional (RNF)

Como o sistema deve se comportar.

RNF02 — Desempenho
A autenticação deve responder em até 2s em 95% dos casos.

Anti-exemplo: "Ter botão azul" (isso é decisão de UI, não RNF).

Regra de Negócio (RN)

Condições/obrigações do domínio.

RN07 — Maioridade
Para cadastrar, o usuário deve ter 18 anos ou mais.

Anti-exemplo: "Usar PostgreSQL" (isso é decisão técnica).

2) Exemplos

Exemplo RF (com critérios)

Título: RF03 — Recuperar senha
Descrição: O sistema deve permitir que o usuário redefina a senha via link enviado por e-mail.
Critérios de Aceitação:
- Enviar e-mail com link válido de 30 minutos.
- Link só pode ser usado uma vez.
- Registrar data/hora da solicitação.

Exemplo RNF (com verificação)

Título: RNF05 — Segurança de Senhas
Descrição: Senhas devem ser armazenadas com hash bcrypt e salt.
Como validar:
- Verificar algoritmo no código/README.
- Checar política de complexidade (mín. 8 chars, maiúscula, número).

Exemplo Regra de Negócio

Título: RN12 — Limite de tentativas
Descrição: Após 5 tentativas inválidas, bloquear login por 15 minutos.
Observações:
- Reset do contador após sucesso.
- Logar IP e horário.

3) GitHub — fluxo

  1. Criar labels mínimas: RF, RNF, Regra-de-Negócio, Prioridade-Alta, Backend, Documentação.
  2. Criar milestone: Requisitos.
  3. Para cada RF/RNF/RN, criar uma Issue com descrição e critérios.
  4. Atribuir labels e vincular ao milestone Requisitos.
  5. Abrir uma branch (ex.: doc/requisitos-aula03), editar os arquivos em /docs e abrir um PR.
  6. No PR, referenciar Issues resolvidas com Closes #N (fecha automaticamente ao dar merge).

Modelo de Pull Request

# Descrição
Breve resumo do que foi feito.

# Issues Relacionadas
Closes #ID (se este PR concluir totalmente a issue)
Relates to #ID (se apenas relacionar)

# Checklist
- [ ] Atualizei /docs/requisitos.md
- [ ] Criei as Issues com labels e milestone
- [ ] Testei os critérios de aceitação (quando aplicável)

4) Atividade

Objetivo: produzir um pacote mínimo de documentação + organização no GitHub para encerrar o milestone Requisitos.

Roteiro cronometrado

  1. Levantamento em dupla/individual: listar ao menos 4 RF, 2 RNF e 2 Regras de Negócio.
  2. Formalização: escrever no /docs/requisitos.md (ou equivalente) com critérios de aceitação/verificação.
  3. GitHub: criar Issues para cada item, aplicar labels e vincular ao milestone Requisitos.
  4. PR: abrir um PR a partir da branch doc/requisitos-aula03 referenciando pelo menos 2 Issues com Closes #

Entrega esperada

  • /docs/requisitos.md atualizado com ≥ 4 RF, ≥ 2 RNF, ≥ 2 RN.
  • Issues criadas para cada item, com labels e milestone Requisitos.
  • 1 PR aberto equipe com Closes # para pelo menos 2 Issues.

5) Dicas e armadilhas comuns

  • Não confundir requisito com tarefa técnica (ex.: "criar tabela").
  • Evite RF vagos (ex.: "gerenciar usuários"). Prefira ações específicas.
  • RNF devem ser mensuráveis (tempo, taxa de erro, compatibilidade).
  • Use Closes # apenas quando o PR concluir totalmente a Issue.
