Diagrama de Classes do Domínio

1. Conceito

O Diagrama de Classes do Domínio é uma representação conceitual dos principais elementos de um negócio e de seus relacionamentos.

Ele descreve o domínio do problema de forma independente de qualquer solução tecnológica.

O foco do diagrama de domínio é o significado do negócio, e não a implementação do sistema.

2. Domínio versus Sistema

O domínio representa a realidade do negócio. O sistema representa uma solução tecnológica que apoia esse domínio.

No diagrama de domínio, modelam-se conceitos reais do negócio, e não componentes técnicos.

3. Classes de Domínio

Uma classe de domínio representa um conceito relevante para o negócio. Ela possui identidade própria e significado claro.

Uma classe só deve existir no diagrama se puder ser reconhecida por especialistas do domínio.

4. Atributos

Atributos representam informações essenciais de uma classe de domínio. Eles descrevem características importantes do conceito modelado.

Nesse nível, atributos não devem refletir detalhes técnicos ou estruturas de banco de dados.

5. Operações

Operações representam comportamentos do negócio. Elas descrevem ações que fazem sentido dentro do domínio.

Não devem representar chamadas técnicas ou responsabilidades de infraestrutura.

6. Relacionamentos

Relacionamentos indicam como os conceitos do domínio se conectam. O relacionamento mais comum é a associação.

Uma associação indica um vínculo estrutural e permanente entre conceitos do negócio.

7. Cardinalidade

Cardinalidade define quantas instâncias de uma classe podem se relacionar com instâncias de outra classe.

Ela expressa regras do negócio, não decisões técnicas.

8. Navegabilidade

Navegabilidade indica quais classes conhecem ou referenciam outras classes.

No domínio, ela representa responsabilidades conceituais, e não acessos técnicos.

9. Generalização (Herança)

A generalização representa uma relação de especialização. Um conceito herda características de outro dentro do domínio.

Seu uso deve refletir a realidade do negócio, e não conveniência de implementação.

10. Classes Abstratas

Classes abstratas representam conceitos genéricos que não existem isoladamente.

Elas servem para agrupar características conceituais comuns.

11. Linguagem Ubíqua

O diagrama de domínio deve usar a linguagem adotada pelo negócio.

Os termos do diagrama devem ser os mesmos usados nas conversas, documentos e regras do domínio.

12. O que não entra no diagrama de domínio

  • Componentes técnicos
  • Classes de infraestrutura
  • Detalhes de banco de dados
  • Frameworks e bibliotecas
  • Conceitos exclusivos de implementação

13. Objetivo do Diagrama de Domínio

Promover entendimento comum entre equipe técnica e especialistas do negócio.

Servir como base para decisões futuras de design, implementação e modelagem de dados.

Regra fundamental:
Se o diagrama não puder ser explicado sem falar de tecnologia, ele não representa corretamente o domínio.
